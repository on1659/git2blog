# 팰월드 서버 패널, “켜고 끄는 툴”에서 “운영 콘솔”이 되기까지

팰월드 서버 패널을 처음 만들 때는 생각이 단순했다.  
웹에서 서버 켜고 끄기만 되면 충분하다고 봤다.

근데 실제 운영은 버튼 두 개로 절대 안 끝난다.

재시작을 눌렀는데 프로세스가 안 죽는다.  
공지 메시지는 보냈는데 종료 타이밍이 서버 상태랑 안 맞는다.  
운영자는 “꺼졌다”라고 믿었는데 실제로는 좀비 프로세스가 남아 있다.  
유저는 “방금 뭐가 된 거지?”를 반복해서 겪는다.

그때 깨달았다.

이건 제어 도구를 만드는 일이 아니라,  
실패를 다루는 도구를 만드는 일이라는 걸.

처음 버전은 성공 경로만 있었다.  
Start 누르면 켜지고, Stop 누르면 꺼지는 경로.

근데 운영에서 중요한 건 성공 경로가 아니다.  
문제가 생겼을 때 다음 행동이 자동으로 이어지는지다.

그래서 종료 로직부터 다시 짰다.  
한 번에 끝내는 방식이 아니라 단계형으로 바꿨다.

공지 먼저 보내고,  
대기 시간을 두고,  
카운트다운을 보여주고,  
실제로 죽었는지 재확인하고,  
안 죽었을 때만 폴백 강제 종료로 넘어가게.

이 순서가 별거 아닌 것 같아도 운영 체감은 완전히 달랐다.

예전엔 “왜 또 안 꺼져?”가 먼저였는데,  
지금은 “지금 어느 단계에서 멈췄는지”가 먼저 보인다.

불안이 줄면 판단 속도가 빨라진다.

여기서 제일 체감 좋았던 건 종료 대기 시간 웹 설정이었다.

운영 중에 대기 시간을 바꾸려고 파일 열고 설정 수정하고 재시작하는 건 너무 느리다.  
특히 유저가 이미 접속해 있는 시간대엔 더 그렇다.

그래서 1~300초 범위를 웹에서 바로 조절하게 만들었다.

사람 없을 땐 짧게.  
피크 타임엔 길게.  
이벤트 직전엔 보수적으로.

같은 서버인데 운영자의 스트레스가 확 줄었다.

운영 도구는 결국 “수정 속도”가 생명이다.

다음으로 손댄 건 설정 UX였다.

설정 항목이 늘어날수록 사람은 틀린다.  
그리고 급할수록 더 틀린다.

그래서 검색을 붙이고,  
공식 문서에 없는 항목은 따로 표시하고,  
카테고리를 운영 시나리오 중심으로 재정리했다.

접속·채팅, 사망·리스폰, 성능, 편의·이벤트.

이 분류는 디자인 취향 문제가 아니었다.  
장애 상황에서 원하는 항목을 10초 안에 찾게 만드는 구조였다.

운영에서 10초는 진짜 길다.  
특히 유저가 기다리고 있을 때는 더 길다.

로그도 정리했다.

콘솔에만 지나가던 로그를 파일로 남기고,  
운영 중 판단에 필요한 로그를 일관된 위치에서 보게 했다.

문제 발생 시 “재현”이 안 되면 개선이 불가능하다.  
로그는 단순 기록이 아니라 회복 도구다.

여기까지가 안정성 절반이었다.

나머지 절반은 의외로 다른 데 있었다.

“그래서 지금 누가 얼마나 즐기고 있지?”

서버를 안정화한 뒤에 항상 이 질문이 남았다.

운영자는 대체로 감으로 말하게 된다.  
요즘 사람 많다, 밤이 붐빈다, 누구는 자주 온다.

근데 감은 빠르지만 부정확하다.  
그리고 논쟁이 생기면 설득력이 약하다.

그래서 DB를 붙였다.  
SQLite 기반으로 세션을 기록하게 바꿨다.

누가 언제 들어왔고,  
언제 나갔고,  
총 플레이 시간이 얼마인지,  
일별/유저별로 직접 계산할 수 있게.

이 변화가 생각보다 컸다.

운영 의사결정이 감에서 숫자로 바뀌었다.

공지 시간을 바꾸면 체류시간이 어떻게 변했는지 본다.  
이벤트 요일을 바꾸면 재접속 패턴이 어떻게 움직였는지 본다.  
특정 유저 그룹이 갑자기 줄었으면 그 전후 변경사항을 추적한다.

이전엔 “왠지 그런 것 같음”이었다.  
지금은 “데이터가 이렇게 말함”이다.

커뮤니티 운영에서도 차이가 났다.

핵심 유저를 감으로 찍는 대신 실제 활동량으로 본다.  
신규 유저가 며칠 내 이탈하는지 확인하고,  
온보딩 메시지나 공지 타이밍을 조정한다.

서버 운영은 기술 문제 같지만 사실 사람 문제다.  
DB 통계는 사람을 숫자로 환원하려는 게 아니라,  
사람을 놓치지 않기 위한 관측 장치였다.

기술 스택 자체는 화려하지 않다.

Node 기반 서버,  
REST API 연동,  
SQLite 세션 저장,  
운영자 실수 줄이는 UI.

근데 구현 난이도보다 운영 난이도가 훨씬 컸다.

성공 케이스는 개발자가 금방 만든다.  
실패 케이스는 운영하다가 계속 새로 나온다.

안 꺼지는 경우.  
늦게 꺼지는 경우.  
설정 반영이 꼬이는 경우.  
로그가 애매한 경우.  
통계가 끊기는 경우.

그래서 이 프로젝트는 “기능 추가”보다 “실패 수습”이 중심이었다.

그리고 결국 이게 품질을 만들었다.

이번 작업에서 가장 크게 배운 건 명확하다.

운영 도구의 완성도는 기능 수로 결정되지 않는다.  
복구력으로 결정된다.

버튼 하나 더 만드는 건 쉽다.  
실패했을 때 안전한 다음 행동을 자동으로 안내하는 건 어렵다.

근데 실제 현장에서 가치가 큰 건 항상 후자였다.

앞으로는 여기서 한 단계 더 갈 생각이다.

지금은 통계를 보여주는 단계라면,  
다음은 통계를 기반으로 운영 추천을 주는 단계다.

예를 들어 최근 2주 패턴 기반으로  
공지 적정 시간,  
재시작 안전 구간,  
이벤트 권장 슬롯을 제안하는 식.

운영자는 매번 처음부터 판단하지 않아도 되고,  
패널은 단순 제어 화면을 넘어서 판단 보조 화면이 된다.

서버 운영은 결국 집중력을 계속 소모한다.  
좋은 도구는 기능을 늘리기보다 그 소모를 줄여준다.

이 프로젝트도 그 방향으로 계속 밀어볼 생각이다.

제어 패널에서 운영 패널로.  
운영 패널에서 판단 시스템으로.

작은 차이처럼 보여도,  
서버 품질은 이런 전환에서 만들어진다.

> 운영 패널의 완성도는 성공 경로가 아니라 실패 경로에서 결정된다.
